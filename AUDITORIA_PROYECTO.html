Nomada - Frontend Travel Agency

19 de mayo de 2026 Stakeholder: Carla Frontend-Travel-Agency

1. Resumen Ejecutivo

El proyecto Nomada es un frontend desarrollado en React 19 + Vite 8 para una agencia de viajes. La aplicación se divide en dos áreas principales: una parte pública (búsqueda de destinos, ofertas, checkout) y un panel de administración (gestión de reservas, hoteles, autobuses, conductores, viajes y usuarios).

La arquitectura sigue el patrón Atomic Design y utiliza Tailwind CSS para los estilos. Los datos se consumen de un backend REST en http://localhost:8080.

9. Recomendaciones Prioritarias

9.1 Críticas (Antes de Entrega)

  1. Integrar Cloudinary en formularios
    Modificar HotelForm.jsx, BusForm.jsx, DriverForm.jsx y TravelForm.jsx para incluir un botón de subida de imagen que use cloudinaryService.upload(). Mostrar preview de la imagen subida.
  2. Agregar validaciones de negocio en CheckoutPage
    Verificar availablePlaces >= pasajerosSeleccionados antes de cotizar. Verificar startDate > hoy antes de permitir reserva. Mostrar mensaje claro si hotel o bus están completos.
  3. Eliminar doble useEffect en BusesCRUD
    Quitar la línea 38: useEffect(() => { load() }, [])
  4. Usar constante API en DashboardPage
    Reemplazar const API_BASE = 'http://localhost:8080/api' por import { API } from '../constants/api'

9.2 Importantes

  1. Unificar cliente HTTP
    Opción A: Usar apiClient (axios) en todos los servicios. Opción B: Eliminar apiClient y mover la lógica de interceptor 401 a un helper compartido.
  2. Persistir estado de checkout
    Usar URL params (/checkout?travelId=123) en lugar de location.state. O implementar un Context para el estado del checkout.
  3. Ampliar cobertura de tests
    Tests de integración para CheckoutPage. Tests de CRUDs principales. Tests E2E del flujo completo.
  4. Limpiar console.error con emojis
    Reemplazar console.error('❌ Error...') por logging estructurado o eliminar en producción.

9.3 Mejoras Opcionales

  • Usar librería jwt-decode en lugar de decodificación manual
  • Agregar validación de email con regex en AuthPage
  • Agregar campo de experiencia en DriverForm
  • Separar servidor Express (server.js) a repositorio backend independiente
  • Agregar loading states en botones de submit de formularios
  • Implementar debounce en buscadores

10. Conclusión

El proyecto Nomada tiene una base arquitectónica sólida con buena separación de responsabilidades, estructura Atomic Design bien implementada y CRUDs funcionales para todas las entidades requeridas.

Puntos Fuertes

  • Arquitectura limpia y bien organizada
  • CRUDs completos para todas las entidades
  • Componentes reutilizables con Atomic Design
  • Responsive design con Tailwind CSS
  • Tests unitarios para componentes base
  • Manejo de autenticación con JWT

Puntos Débiles

  • Cloudinary sin integrar en formularios (requisito obligatorio)
  • Validaciones de negocio faltantes en checkout
  • Inconsistencia de cliente HTTP (axios configurado pero no usado)
  • Cobertura de tests insuficiente para páginas y flujos críticos
  • Checkout frágil dependiente de location.state

Veredicto

El proyecto está en un estado funcional pero incompleto respecto a los requisitos del briefing. Con las correcciones prioritarias señaladas (Cloudinary, validaciones de negocio, limpieza de código duplicado), el proyecto puede cumplir con todos los criterios de aceptación para la entrega final.
